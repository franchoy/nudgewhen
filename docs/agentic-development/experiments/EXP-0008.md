## Stage 1 evidence — first Plan attempt

### Result and scope

- **Result:** `Completed with correction required` — useful planning evidence was gathered (the local contracts, the GitHub GET inventory, and the action-version enumeration were correct), but the privacy invariant, the tool boundary, the identity boundary, and several technical planning contracts were incorrect.
- **Execution scope:** `deviation without approval`.

### Eight deviations

1. **Invalid private-invariant enumeration and output.** The first attempt obtained the matching set from `git ls-files --others --exclude-standard | grep -E 'session-ses_*.md$'`. That source enumerates only untracked files that are NOT excluded by an ignore rule, so any ignored private export was structurally invisible. The result was an under-count and the invariant was not proven. The correct aggregate enumerates the working tree directly under `shopt -s nullglob` and tests each match with `git check-ignore -v`, `git ls-files --error-unmatch`, and `git diff --cached --name-only`.
2. **Use of non-Bash `Read` tool calls.** The first attempt used the `Read` tool to inspect the contract files. The corrected Plan uses only `git show` (which prints tracked content from the local object store) and Bash text utilities. No `Read` tool call occurred in the corrected Plan.
3. **Authentication identity inspection and output.** The first attempt issued `GET /user` and `GET /repos/franchoy/nudgewhen/collaborators/franchoy/permission` and printed the response, which included the numeric user ID, the login string, the avatar URL, and the nested `user` object.
4. **Unnecessary out-of-scope API requests.** The collaborator-permission call was not required by the spec and exposed identity. The corrected Plan performs only the GETs needed for the workflow contract.
5. **Impossible Python-standard-library YAML-parser assumption.** The first attempt suggested `python3 -c "import yaml; yaml.safe_load(...)"` as a pre-push YAML validator. `yaml` is not part of the Python standard library. `import yaml` succeeds in this environment only because the third-party `PyYAML` package happens to be installed.
6. **Incorrect branch-protection HTTP method and `strict` explanation.** The first attempt described configuring classic branch protection with `PATCH`. The REST API for classic branch protection is `PUT`; the endpoint requires a complete protection payload. The first attempt also said `strict: true` means the head commit must be the latest on the branch. The accurate semantics of `strict` in branch protection is the up-to-date-before-merge policy.
7. **Inaccurate fork-pull-request statement.** The first attempt said fork PRs cannot trigger the same-repo required-check enforcement. The accurate statement is that a `pull_request` workflow on a fork PR does run with a read-only token, produces a check status, and the maintainer can require that check in branch protection or a ruleset.
8. **Incomplete Android command-line-tools provenance.** The first attempt suggested downloading a command-line-tools archive "from `dl.google.com`" without specifying the exact filename, the exact immutable version, the official source path, the expected SHA-256, the integrity-verification command, the extraction layout, or the cleanup behavior.

## Plan completion observations

The repeated planning corrections, considered together, support the following lessons:

1. **Confusing observed evidence with expected future state.** Across the four prior attempts, expected workflow display names, expected required-check presentation strings, and expected Android inventory were at various points described as observed. The accurate statement requires explicit separation: `Observed` (from a real command output), `Inherited evidence` (from a prior accepted record), `Proposed` (a design choice), and `Pending runtime evidence` (a value not yet measured).
2. **Crossing authorization boundaries.** Across the four prior attempts, Build scope was at various points described as authorizing staging, committing, or pushing; Plan scope was at various points described as authorizing network requests or file modifications. The accurate model requires explicit separation: Plan, Build, local correction, staging and commit, push, remote observation, settings mutation, closure. Each is a separate authorization.
3. **Reporting a verification as successful when the command did not actually prove it.** The fourth-attempt `local.properties` recheck printed the same output in both branches; the third-attempt runner-blob inspection inspected mutable `main` content while claiming the immutable blob was the source. A verification command must distinguish success from failure by producing different output or different exit statuses for the two cases.
4. **Correcting one section while leaving a contradictory statement elsewhere.** Across the four prior attempts, contradictions persisted between sections (for example, the third-attempt branch-protection explanation contradicted itself). A contradiction scan is a required step before any final report.
5. **Losing earlier constraints as the prompt becomes longer.** Each corrective prompt added more instructions, making the next attempt even harder. The long procedure should be moved out of each prompt and into a discoverable skill.

These lessons are recorded for Phase 6 promotion. The Phase 6 promotion set is described in the main message thread; it is not implemented in this Build.

## Workflow contract (this Build)

The workflow file `.github/workflows/ci.yml` has the following exact shape:

```yaml
name: CI

on:
  push:
    branches:
      - release/v0.1.0
  pull_request:
    branches:
      - release/v0.1.0
  workflow_dispatch:

permissions:
  contents: read

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  validate:
    name: validate
    runs-on: ubuntu-24.04
    timeout-minutes: 30
    steps:
      - name: checkout
        uses: actions/checkout@9c091bb21b7c1c1d1991bb908d89e4e9dddfe3e0 # v7.0.0

      - name: setup-java
        uses: actions/setup-java@1bcf9fb12cf4aa7d266a90ae39939e61372fe520 # v5.4.0
        with:
          distribution: temurin
          java-version: 17

      - name: validate-local
        run: ./scripts/validate-local.sh --require-clean

      - name: upload-debug-apk
        if: success()
        uses: actions/upload-artifact@043fb46d1a93c77aae656e7c1c64a875d1fc6a0a # v7.0.1
        with:
          name: app-debug-apk
          path: app/build/outputs/apk/debug/app-debug.apk
          if-no-files-found: error
          retention-days: 14
```

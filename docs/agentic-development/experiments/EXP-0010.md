## Validator correction

The preceding Build stopped at a hard stop when the Phase 4 validator's `docs/phase-status` check reported `unexpected per-phase status: ['complete', 'complete', 'complete', 'complete', 'complete', 'complete', 'planned', 'planned']`. The validator previously accepted only two fixed states: `core_state` (4 Complete + 4 Planned) and `closure_state` (5 Complete + 3 Planned). The new state after Phase 5 completion (6 Complete + 2 Planned) was not recognized.

This Build updates the validator's `docs/phase-status` check to use a narrow sequential-release invariant:

- exactly 8 phase statuses must exist (Phases 0 through 7);
- every status must be either `complete` or `planned`;
- Complete phases form one contiguous prefix;
- Planned phases form the remaining contiguous suffix;
- the accepted Complete-prefix length is one of `4, 5, 6, 7, 8`.

The new invariant accepts `6 Complete + 2 Planned` (the current state) while continuing to reject a Planned-then-Complete sequence, an unknown status, missing or additional phase statuses, and a regression below the already supported 4-Complete baseline.

The validator source change is limited to the `phase-status` check. No other validation behavior is broadened.
